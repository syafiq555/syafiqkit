## 1.64.12

- **ship**: Three fixes from a real production ship that the skill's own steps walked into. (1) **Task-doc deploy state.** Step 2.4's staleness gate told the session to fix stale docs before committing — so it wrote "NOT yet on production / 🚢 in flight" into the task doc *while running `/ship`*, committed it, and needed a second commit to undo it once the deploy landed twenty minutes later. The gate never said *which* truth to write, and mid-ship the state that's true right now is guaranteed wrong by the time anyone reads it. The trap sprang hardest immediately after correctly catching a *false* "deployed via rsync hotfix" claim: correcting it felt obviously right, but the correction went to the transient midpoint instead of the outcome. (3) **Blast radius.** Step 3's "what rides along" check only looked for migrations; a commit subject-lined `chore: prune stale agent memory` carried a live payout-schedule change (agencies paid 2×/week → 4×/week) with no changelog entry. Read the diff, never the commit subjects.

## 1.64.5

- **merge-task-docs**: the back-reference scan shipped `rg -rn` in three literal copy-pasteable commands — and `rg` has **no recursive flag** (it always recurses), so `-r` is `--replace`: the command printed every match with the searched path *substituted away*, exit 0, no error. Step 3 scans for back-references and Step 6's exit condition is literally "zero results = done" — with the source docs already deleted by Step 5. A corrupted scan therefore reads as a clean all-clear and the merge finishes with dangling pointers. Switched to `grep -rn` (in ugrep, `-r` genuinely *is* recursive), and the final scan now carries a must-hit control line, because a zero from a broken search is indistinguishable from a genuinely clean tree.
- **read-summary**: discovery told the reader to use the `Grep`/`Glob` tools but said nothing about the fallback when those tools aren't available in a session — so the fallback became Bash `rg`, with the `-r` footgun and no warning. Found the hard way: a discovery search typed `rg -ril "payout"` intending *recursive, case-insensitive, list-files*, and got back output with every "payout" replaced by "il" (`-r` consumed the `l` as its replacement string). The rule already existed, loudly, in global CLAUDE.md — and was broken anyway, because at discovery time the mental frame is "I'm finding a doc", not "I'm writing a shell command". So the fix isn't another copy of the rule: it names `grep` as the mandated fallback (safe by construction) and gives the *tell* — your search term is missing from its own output.

## 1.64.1

- **done**: Step 5's gate now catches **the signal you CAUGHT**, which is the one it kept missing. If you declined to follow a skill's step, worked around it, or told the user "this step doesn't apply here" — that IS the defect, and it's the hardest one to notice precisely because it feels like a win ("I handled it"), so it gets filed as competence instead of a finding. The gate's wording was calibrated entirely against false positives (*"most runs have none"*, *"never manufacture one"*, *"a thin patch is worse than no patch"*) with nothing guarding the other direction. Found the hard way: this session hand-rolled a correct branch chain *because* `/ship`'s Step 3 was wrong, then skipped Step 5 anyway. New check: *did I deviate from any skill's written instructions this session, and why?* — a deviation with a good reason is a skill that needs the reason written into it.
- **ship**: Step 3 no longer assumes **`git push` on the current branch IS the deploy**. It isn't, on any project that promotes through a chain (`master → staging → production`), and the failure is silent in the worst way: pushing `master` can deploy *nothing* (production needs a forward-merge plus a manual CI gate) or deploy to *staging* while the skill reports "shipped to production" — CI goes green either way, so the verify step's green tick confirms the wrong thing. Worse, the deploy branch differs **per repo**: in a two-repo ship one repo's `master` was its staging branch while the other's deployed nothing at all, so a single `git push` per repo produced two different wrong outcomes at once. Step 3 now requires establishing the deploy branch first (from `CLAUDE.md`/`CLAUDE.local.md`, confirmed against the repo), recognising a forward-merge chain as a *merge*, not a push (`--ff-only` fails on diverged branches), checking for a manual gate, and surfacing any migrations riding along — a one-way door that belongs in the user's decision, not a silent side effect. If the chain isn't documented and can't be inferred: **ask**, because pushing to a deploy branch is outward-facing and hard to reverse.

## 1.64.0

- **done**: new **Infra-only mode** — the mirror image of docs-only. When the diff is entirely deploy/build/CI plumbing (CI workflows, `docker-compose*.yml`, `Dockerfile`, `next.config.*`, nginx/env), run the **reviewer only**: there's no logic to DRY and no user journey, but infra needs review *badly* because it fails **silently in production** (a `docker compose exec` swallowing a heredoc's stdin; `nginx -s reload` re-reading an orphaned inode; an undeclared `NEXT_PUBLIC_*` build arg dropped at compile time — all pass lint/typecheck/tests). Size-independent: the trigger is file KIND, not count. Also warns against blanket-applying a fix to every occurrence of a pattern — two call sites of the same command can need opposite treatments. Exception: a compose change that flips a feature flag ON is not infra-only (user-facing capability → product reviewer).
- **read-summary** — four fixes, all from this skill running and *still* letting the session get things wrong:
  - **Trigger now names deploy/go-live questions** ("if we deploy staging to production, what do we need to do?"). These were a blind spot: they name no feature and report no bug, so the old description didn't match — yet prerequisites, blockers, and env state live in task docs and *nowhere in the code*. Now a PRIME trigger, not an edge case.
  - **New: what the doc is authoritative FOR.** A task doc is authoritative for *decisions* and *gotchas*; it is **not a live-state oracle** (what's in prod's DB/`.env`/a bucket right now decays the moment anyone touches a server). The trap is that a well-read doc makes you feel fully grounded, so you answer a live-state question from a weeks-stale snapshot with total confidence. If the answer depends on a running system, **measure it** — the doc's claim about prod is a hypothesis to test, and when they disagree the live system wins.
  - **Never infer a REMOTE system's state from a LOCAL file's absence.** "Prod's `.env` has no `AWS_*`" means *this machine isn't wired to S3* — it says nothing about whether the bucket exists. Ask the remote system with a call that discriminates (`HeadBucket`: 403 exists-denied vs 404 absent).
  - **Sibling-repo rule promoted to its own Read Order step**, gated on the QUESTION rather than on which files you'll edit. It had been a sub-bullet of the CLAUDE.md tree-walk — which fires per-directory-you-edit — so a read-only planning question skipped it precisely when it mattered most. (Sibling `CLAUDE.md`/`CLAUDE.local.md` never auto-load: the harness walks from the working dir only, and it hides itself — the current repo's files *did* load, so context feels complete while you re-derive facts the sibling already documents.)
- **task-summary**: §6 back-reference reconciliation now also greps the **defect's own vocabulary**, not just the feature's. A doc describing a bug you just fixed is stale in a way a feature-name grep structurally cannot find — it may never name the feature, only the defect. Higher severity than a stale status: a stale *status* is merely out of date, a stale *gotcha* actively misleads the next session into re-fixing a solved bug or routing around a non-problem.
- **done** ⇄ **update-plugin**: `/done` gains a **conditional Step 5** that closes the loop — a session that exposes a skill defect now feeds it back into the plugin instead of losing it. Two gates, both required: (1) a real skill signal exists (most runs have none — that's expected; merely *using* skills successfully is not a signal), and (2) you own the plugin, verified by probing whether `~/.claude/plugins/syafiqkit` is a git checkout of `syafiq555/syafiqkit`. `update-plugin` gains the mirror gate as **Step 0**, so a direct invocation is safe too.
- **Consumer findings now reach the maintainer automatically** — `update-plugin` gains an upstream mode: on a non-owner install it still scans, then **offers to file the finding as a GitHub issue** (`gh issue create --label skill-feedback`), which notifies the maintainer instantly. No secret ships in the plugin and no server is needed — the post authenticates as the *user's own* GitHub identity, so nothing is extractable from a public plugin. ⚠️ It **asks first**: an unprompted outbound post under someone's own name is a surprise action, and silent phone-home telemetry is not acceptable. Declined or no `gh` → falls back to a copy-pasteable report. New `skill-feedback` label on the repo makes these filterable.
  - Patching a consumer's install would be *harmful* anyway: `claude plugin update` overwrites the directory, so the edit silently vanishes and diverges from upstream meanwhile. Write-permission is not the test — whether the edit *survives and belongs* is.
  - Gotcha captured: `gh label list --search <name>` returns **empty for a label that exists** — verify with `gh api repos/OWNER/REPO/labels/<name>`, never conclude "missing" from the search.
- Captured after a two-repo production-go-live planning session: the sibling repo's CLAUDE.local.md already documented the prod S3 bucket + both IAM keys, and a from-scratch re-derivation got it wrong; and fixing the `nginx -s reload` bug left three separate docs still asserting it as live (one of them never mentioning the feature at all).
