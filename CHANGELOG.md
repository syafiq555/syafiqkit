## 1.102.0

- **done, explore-delegation, two-tier-condense, tackle, agent-architecture (D32)**: corrected a factually wrong rule the plugin asserted in two places and contradicted in a third. `done`'s rules table and `explore-delegation.md` claimed `run_in_background: false` makes an `Agent` call block; `two-tier-condense.md` said the opposite ("agents run in the background by default") — and *it* was right. [Official docs](https://code.claude.com/docs/en/sub-agents): *"As of v2.1.198, subagents run in the background by default. Claude runs a subagent in the foreground when it needs the result before continuing."* [Issue #69691](https://github.com/anthropics/claude-code/issues/69691) (OPEN) adds that `false` is honoured in child sessions and **ignored in top-level interactive ones**. The prior CHANGELOG entry (1.x, "pass it explicitly") had already escalated this in the wrong direction — the rule was wrong, not weakly worded, so escalating it compounded the error.
- **The rule that was actually being violated**: parallelism is **every `Agent` call in ONE assistant message**. No flag substitutes for it; one-per-message serialises regardless of `run_in_background`. Conflating the two meant a session could set the flag correctly, emit calls one at a time, and believe it had a parallel batch. `done` Step 1, `tackle` Step 3, and `two-tier-condense` now state the block *is* the parallelism.
- Skills must now **tolerate an async return** from any delegated `Agent` — a step that reads a result "on the next line" is unsafe by construction. Results arrive as `<task-notification>` and are never lost: never poll (`sleep`/`ScheduleWakeup`/`TaskOutput`), never Read an agent's `.output` file (full subagent JSONL — overflows context).
- Recorded, not adopted: frontmatter `background: true` is the only *documented* per-agent control, and it forces background **on** — there is no documented way to force foreground at any level.

## 1.98.0

- **two-tier-condense, condense-task-doc, condense-claude-md**: fixed the draft-verify baseline. The shared reference carried the correct rule as a bare parenthetical ("or against the pre-draft snapshot if the file was already dirty") while both consuming skills stated `git diff HEAD -- <file>` as an unqualified ⚠️ imperative — so the louder, closer instruction won and the caveat never fired. On a doc carrying uncommitted edits, HEAD is the wrong baseline: the *other writer's* changes surface as `-` lines and read exactly like the drafter rewriting a protected block, which is the precise failure the verify step exists to catch. The shared reference now owns the baseline rule with its tell (a `-` line whose text you recognise as an edit made earlier this session = wrong baseline, not fabrication); both consumers point at it instead of restating a contradicting default.
