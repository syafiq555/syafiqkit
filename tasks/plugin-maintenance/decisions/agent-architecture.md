# Plugin Maintenance ‚Äî Agent Architecture Decisions

Decisions about how generated project agents (`.claude/agents/*.md`) inherit conventions, delegate to sibling skills, and reliably invoke them.

---

### D1 ‚Äî Project-Specific Agents via Prompt Injection ‚Äî committed ‚Äî 2026-02

**Problem**
Agents don't inherit CLAUDE.md, so project conventions aren't applied when a spawned agent edits code or reviews a diff.

**Decision**
Chosen: bake conventions into agent system prompts via injection markers, generated by `agent-setup`.
```
/update-claude-docs ‚Üí Updates CLAUDE.md ‚Üí invokes agent-setup ‚Üí creates/updates .claude/agents/*.md
/done ‚Üí checks .claude/agents/{code-simplifier,code-reviewer}.md ‚Üí project agent OR fallback to external
```
```markdown
<!-- INJECTED FROM CLAUDE.md -->
[Project-specific rules extracted from CLAUDE.md]
<!-- END INJECTED -->
```

**Rejected**
- Runtime CLAUDE.md parsing per agent call. Why not: no persistence, re-parses every invocation, agents stay generic between runs.

**Consequences**
- No runtime CLAUDE.md parsing needed; agents are project-aware and portable with the repo; gotchas accumulate cumulatively in the agent file itself.
- Gotcha: `/done` blocks if a template omits a needed tool (e.g. `code-reviewer` needs `Bash(git:*)` for `git diff`) ‚Äî add to the agent's tools list, don't work around it.
- Gotcha: a hand-written agent dropped into `.claude/agents/` isn't reproducible ‚Äî always register via `agent-setup` (SKILL.md registry + `templates/<name>.template.md`), never hand-write a canonical agent.

**Status**: committed ¬∑ **Reversible**: yes

### D4 ‚Äî Orchestrator Skills Delegate, Never Inline a Sibling's Procedure ‚Äî committed

**Problem**
`done` Step 3 used to contain a full copy of `update-claude-docs`' signal-scan procedure. Two copies drift, forcing "don't double-write" hedges to reconcile them.

**Decision**
Chosen: an orchestrator names the sub-skill and delegates; it never reproduces the sub-skill's procedure inline.

**Rejected**
- Keeping both copies in sync manually via hedge comments. Why not: hedges are a symptom of duplication, not a fix ‚Äî they still drift.

**Consequences**
- Applied again in `agent-setup`'s generated agents: templates now carry `Skill` in `tools:` and name `/read-summary` as canonical for task-doc discovery, with a short inline fallback only for when the skill can't be invoked (D14).
- Gotcha: correct wiring (tool + instruction) doesn't guarantee the model reliably calls the sibling skill ‚Äî see D15.

**Status**: committed ¬∑ **Reversible**: yes

### D14 ‚Äî Generated Agents Invoke `/read-summary`, They Don't Reimplement It ‚Äî committed

**Problem**
Agent templates were hand-copying `read-summary`'s Glob-`tasks/**/*.md`+Grep-vocabulary logic into each Bootstrap ‚Äî which drifts (sibling-repo path, synonym rule, `Related:`-follow all restated, all going stale independently).

**Decision**
Chosen: every task-doc-consuming agent (all but `claude-md-pruner`, which is CLAUDE.md-only) carries `Skill` in its `tools:` and names `/read-summary` as canonical, with a short inline fallback for when the skill can't be invoked.

**Consequences**
- Same principle as D4, applied to `agent-setup`'s output specifically.
- Caught the drift's tell along the way: `Explore`'s `description:` claimed it "reads task docs" while its Bootstrap never mentioned them (frontmatter drives routing, body drives behavior ‚Äî disagreement = under-delivery). Patched SKILL.md (Key-rules + Step-5 verify checkbox) AND all 5 templates so a regen doesn't drop it.

**Status**: committed ¬∑ **Reversible**: yes

### D21 ‚Äî Step 5's Checklist Requires a Command Per Item, Not a Re-Read ‚Äî committed ‚Äî 2026-07-12

**Problem**
A live `/agent-setup` run on an existing 6-agent project re-read all six files, judged them "well-established" from that skim, and reported the Step 5 checklist as satisfied. The user pushed back ("check properly"); a literal grep against the same files immediately found 2 failing items (`Skill` tool + `/read-summary` wiring missing from 5 agents per D14; `disallowedTools: [Write, Edit]` missing from `Explore`/`Plan` per the naming-exception note) that the skim had missed entirely.

**Decision**
Chosen: Step 5's checklist is prefaced with an explicit instruction that each item is a command to run against current file content, not a fact to recall from having read the file earlier in the session. The "Agents exist" row in Step 1 also no longer permits skipping Step 5 ‚Äî an established-looking agent set still gets the full checklist.

**Rejected**
- Trusting a prior in-session file read as sufficient verification. Why not: reading a file for "does this look right" and grepping it for "does this literal string exist" are different operations with different failure rates ‚Äî the session's own before/after is the proof (same six files, skim said pass, grep said fail on 2/17 items).

**Consequences**
- Distinct from D15: D15 is about whether a *generated agent* reliably calls `/read-summary` at runtime. D21 is one layer up ‚Äî whether the model *executing agent-setup itself* reliably runs its own verification checklist, or silently substitutes a skim.
- General pattern, not `agent-setup`-specific: any skill with a Step-N "verify" checklist is exposed to the same substitution unless the checklist text itself forecloses it.

**Status**: committed ¬∑ **Reversible**: yes

### D24 ‚Äî A Self-Caught Deviation From a Skill's Own Instructions Is a Reportable Signal ‚Äî committed ‚Äî 2026-07-13

**Problem**
`done` Step 5's gate ("does a real skill signal exist?") was calibrated entirely against false positives ‚Äî wording like "most runs have none" and "never manufacture one" ‚Äî with nothing guarding the opposite failure. A live session hand-rolled a correct branch chain because `/ship`'s Step 3 wrongly assumed `git push` on the current branch was the deploy, then almost skipped Step 5 anyway: catching and working around the defect in the moment felt like competence, not a finding, so it nearly went unreported.

**Decision**
Chosen: `done` Step 5 now asks explicitly ‚Äî *did I deviate from any skill's written instructions this session, and why?* A deviation with a good reason is a skill that needs the reason written into it; the gate treats a self-caught workaround as equivalent to a user-flagged misfire.

**Rejected**
- Leaving the gate as "does something feel like a bug" and trusting the model to generalize to self-caught cases. Why not: the same session that caught the deviation is the one that nearly filed it as a non-event ‚Äî the moment of catching it is exactly when it stops looking like a defect.

**Consequences**
- Directly produced the `ship` Step 3 fix in the same session (D-equivalent fix, not yet numbered as its own ADR ‚Äî see CHANGELOG 1.64.1): deploy-branch identification now required before any push, recognizing forward-merge chains as merges (not pushes) and surfacing manual gates + migrations riding along.
- Distinct from D21: D21 catches a *checklist skimmed instead of run*; D24 catches a *defect the model already fixed in place*, which is arguably easier to miss because there's no failing check to notice ‚Äî only a decision not to mention it.

**Status**: committed ¬∑ **Reversible**: yes

### D15 ‚Äî Correct Wiring to Invoke a Sibling Skill ‚âÝ the Model Reliably Calling It ‚Äî committed

**Problem**
1.55.0 gave `Explore`/`Plan` the `Skill` tool and named `/read-summary` in Bootstrap (D14), but a real session still skipped the call ‚Äî a free-text prompt that already named exact files/methods for an OAuth flow read as a fully-scoped code trace, so the agent went straight to CLAUDE.md reads instead of discovery.

**Decision**
Chosen: add an explicit counter-instruction to `Explore`/`Plan` templates ‚Äî prompt specificity implies a task doc is MORE likely to exist (the caller's detail came from somewhere), not a reason to skip discovery.

**Rejected**
- Trusting the wiring (tool + instruction) alone as sufficient. Why not: Bootstrap instructions are advisory prose an agent can judgment-call around; wiring doesn't force the call the way a hook or schema constraint would.

**Consequences**
Only `Explore`/`Plan` needed this fix ‚Äî `code-reviewer`/`code-simplifier`/`product-reviewer` trigger off `git diff`, not free-text prompts, so they aren't exposed to the same misjudgment.

**Status**: committed ¬∑ **Reversible**: yes
